## Issue 18: Tag chip-checkbox UI everywhere + chronological time-period sort on Summary

### What to build

Two related changes that affect the tag-input UI in three places and the sort/labeling behavior of the Summary table, plus the server-side contracts, safe-rendering rules, and error-handling that keep both correct under tampering or partial failure:

1. **Unify the tag-selection UI as a chip-checkbox block.** On every page where a user selects tags — the expense entry form, the recurring-template form, the expense edit page, the list-page tag filter, and the Summary tag filter — render every existing tag as a labeled `<input type="checkbox">` styled as a chip, alphabetically sorted (case-insensitive), wrapping to the viewport. Selected chips must be visually distinct from unselected ones. The list-page tag-filter UI (`renderFilterBar` in `expense-list-renderer.tsx`) already implements this pattern with `flex-wrap` / `gap-2` and per-tag checkboxes; use it as the reference and extract a shared component (e.g. `src/components/tag-chip-checkboxes.tsx`) that is reused everywhere.

2. **Inline new-tag input on mutation forms only** (expense create, expense edit, recurring create, recurring edit). On every mutation form, render a small text input adjacent to the chip block that accepts one or more new tag names (comma- or whitespace-separated). New names flow through the existing inline-create confirmation page (same flow used today for new categories and the current CSV tag input). The list-page filter and the Summary-page filter do **not** get the new-tag input — they only operate over existing tags.

3. **Replace the existing tag input mechanism.** The forms currently submit a CSV string via a single `tags` text input enhanced by `public/js/tag-chip-picker.js`. Replace this with:
   - server-rendered native `<input type="checkbox" name="tagId" value="<tagId>">` chips for existing tags,
   - a separate `<input type="text" name="newTags">` for new tag names,
   - a renamed/rewritten `public/js/tag-chip-checkboxes.js` whose only job is to (a) style the chip toggle interaction and (b) optimistically render typed new-tag names as already-selected chips using safe DOM APIs (`textContent`, never `innerHTML`). Optimistic chips are visual only — the submitted form data still flows through the existing native fields. JS init failures must be swallowed (logged via `console.error`) and must not block native checkbox toggling or form submission. Without JS, native checkboxes continue to work and `newTags` is submitted as-is to the confirmation page.
   Update `expense-validators` and the post/confirm handlers (`expense-post-handler.ts`, `expense-confirm-post-handler.ts`, and the recurring equivalents — including `recurring-post-handler.ts` and `recurring-confirm-post-handler.ts` or their project-equivalent names) to read `tagId[]` + `newTags` instead of the CSV `tags` field, applying the **server-authoritative tag input contract** (see _Server contract_ below). Preserve the existing inline-create confirmation flow end-to-end; normalized new tag names must match `^[a-z0-9_-]{1,20}$` (ASCII letters, digits, hyphens, underscores only — no whitespace, no Unicode letters, no other punctuation), with trim/lowercase/dedupe/empty-drop unchanged.

4. **Replace the Summary tag filter `<select multiple>` with the shared chip-checkbox block.** In `build-summary.tsx`, swap the current `<select id="summary-tag-filter" multiple>` for the same shared component used on entry/list, preserving the `name="tagId"` query-string contract (the existing `parseSummaryQuery` already accepts `tagId[]`). Continue to require signed-in access on every affected page and POST handler (entry, edit, recurring create/edit, confirmation routes, list filter, Summary). The list-page tag filter retains its existing AND/OR toggle and `mode=and|or` query-string contract unchanged.

5. **Time-period column sorts chronologically, never alphabetically; labels include the year.** In `src/lib/db/summary-access.ts`, the current implementation sorts rows by `timePeriod` using `localeCompare`, which puts `Apr` before `Feb` and `Apr-Jun` before `Jan-Mar`. Replace this with an internal chronological key carried alongside the rendered label:
   - Month granularity → key `(year, monthIndex 0..11)`; rendered label `Mmm YYYY` (e.g. `Mar 2026`).
   - Quarter granularity → key `(year, quarterIndex 0..3)`; rendered label `Mmm-Mmm YYYY` (e.g. `Jan-Mar 2026`).
   - Year granularity → key `year` (numeric); rendered label `YYYY`.
   Month and Quarter rows are keyed by `(year, index)` so rows from different calendar years are distinct rows and never aggregate together (e.g. `Mar 2025` and `Mar 2026` are separate rows). Apply chronological ordering both in the **default sort** (group columns asc, then time-period chronologically asc by the internal key) and when the user clicks the time-period column header (toggles asc/desc but always uses the internal key, never the rendered label). The category/tag column comparators remain case-insensitive alphabetical; `count` and `total` remain numeric. **No code path in the Summary sort may compare rendered month or quarter labels alphabetically.** When the user clicks `count`, `total`, `category`, or `tag`, ties retain the default group/time ordering (group columns asc, then chronological time-period asc). Validate the `sort` query input against an allow-list of column names and directions; the allow-list is **dimension-aware** — valid sort columns are those present in the current result table for the selected dimension, plus the always-present `count`, `total`, and `timePeriod`. Concretely: with dimension `Category` or `Time only`, `sort=tag` is invalid; with dimension `Tag` or `Time only`, `sort=category` is invalid. Invalid (out-of-dimension or unknown) sort columns and invalid sort directions fall back to defaults rather than erroring.

6. **Cleanup.** Delete `public/js/tag-chip-picker.js` and every `<script src="/js/tag-chip-picker.js">` reference (in `expense-form.tsx`, `recurring-form.tsx`, `build-edit-expense.tsx`, `build-create-recurring.tsx`, `build-edit-recurring.tsx`, `expense-list-renderer.tsx`) once the new component is in place. After cleanup, no rendered page may produce a 404 for `/js/tag-chip-picker.js`. Remove the now-unused `data-tag-chip-picker` attribute and the `tagsCsvMax`-based CSV input field on those forms. The CSV-parsing path in the validators and post handlers can be removed if and only if no other caller depends on it.

7. **Server contract for submitted tags** (applies to mutation forms, the list filter, and the Summary filter; never trust the UI):
   - **`tagId` syntactic validation and caps (pre-DB)**: before any existence lookup, each submitted `tagId` value must syntactically match the project's ID format (ULID — 26-character Crockford base32, **uppercase-only** per the regex `^[0-9A-HJKMNP-TV-Z]{26}$`; lowercase input is rejected as invalid, never silently uppercased — all ids generated and stored in this app are uppercase). Invalid-format values are treated as unknown ids and never reach the database.
     - Mutation forms additionally cap the **raw** submitted `tagId` count (before dedupe) at a small maximum (e.g. 64); exceeding the cap is a recoverable validation error — 65 copies of the same valid id still errors even though dedupe would yield one.
     - List filter and Summary filter requests truncate to the same cap and ignore the remainder rather than erroring (the page still renders).
     - The existing global request-body size limit applies as an outer bound on every mutation submission; the per-field caps here are inner bounds that fail fast with a recoverable validation error before any DB work.
   - **`tagId` existence validation**: after syntactic + cap filtering, dedupe repeated ids, then validate that each remaining id exists in `tag`.
     - Mutation forms: an unknown or stale `tagId` (e.g. tag deleted between page render and submit, or syntactically invalid) renders a recoverable global error on the form with all other field values preserved.
     - List filter / Summary filter: unknown or stale `tagId` values are silently dropped; the page still renders with the remaining valid selections, and the rendered filter UI simply omits the stale ids (no error, no "selected unknown tag" placeholder).
   - **`newTags`** (mutation forms only): split on any run of commas and/or ASCII whitespace, then trim → lowercase → drop empty tokens → enforce `^[a-z0-9_-]{1,20}$` (ASCII letters, digits, hyphens, underscores only — no whitespace, no Unicode letters, no other punctuation) → dedupe case-insensitively. Reject per-token names that exceed 20 chars or contain disallowed characters. Cap raw `newTags` length (e.g. 500 chars) and post-split token count (e.g. 32 tokens); exceeding either is a recoverable validation error. Accepted separator examples that all yield `[foo, bar]` or `[foo, bar, baz]`: `foo,bar`, `foo bar`, `foo, bar`, `foo,,bar`, `foo\nbar`, `foo,  bar , baz` — empty tokens between separators are dropped.
   - **Collision** between `tagId` and `newTags`: if a normalized `newTags` token matches the lowercase name of any existing tag (selected or not), treat it as a selection of the existing tag — do not create a confirmation entry for an already-existing name; do not double-attach.
   - **Confirmation-time race (tags and categories)**: if another request creates a tag (or category) with the same name between the initial submit and the confirmation submit, the confirm handler must reuse the now-existing row (silently attach the tag, or select the existing category) rather than failing with a generic unique-constraint error. The existing category race/collision handling already in the confirmation flow must remain intact — this contract extends it, not replaces it. Other unique-constraint failures surface a recoverable user-facing message that re-renders the confirmation page with all values preserved.
   - **Category input contract (parallel to tags; never trust the UI)**:
     - The submitted `categoryId` (single value, optional only when a new category name is being introduced) is syntactically validated against the same uppercase ULID regex `^[0-9A-HJKMNP-TV-Z]{26}$` before any DB lookup; invalid-format values are treated as unknown.
     - Unknown / stale `categoryId` on mutation forms produces a recoverable global error with all field values preserved (no DB lookup is attempted for the bad id).
     - A submitted new-category name is trimmed → lowercased → validated against `^[a-z0-9_-]{1,20}$`; longer or character-set-violating names produce a recoverable validation error.
     - Collision: if the submitted new-category name matches the lowercase name of an existing category, the confirm flow treats it as a selection of the existing category, not as a creation.
     - Confirmation-time race: if another request creates a category with the same name between the initial submit and the confirm submit, the confirm handler silently reuses the existing row — extending, not replacing, the existing category race/collision handling.
     - Confirmation revalidation re-runs all of the above on every confirmation submit; the canonical confirmation payload includes `categoryId` (or the new category name) and is covered by the HMAC signature described in the next bullet.
   - **Confirmation revalidation (server-authoritative) + tamper detection**: the confirm handler must fully re-run every validation step on the submitted confirmation payload — field shape, character-set, length, caps, `tagId` syntactic check, `tagId` existence, collision normalization, money parsing, date validity, `categoryId` syntactic + existence check, etc. — and must not trust any value carried in hidden fields from the prior step. Hidden inputs are a round-trip convenience only. **Revalidation proves the payload is valid; it does not prove the payload is the same values shown on the confirmation page** (e.g. flipping a hidden amount from `10.00` to a valid `20.00`, or swapping a valid `categoryId` for a different valid one, passes normal validation). The confirmation page therefore carries an HMAC signature over the canonical confirmation payload (sorted keys, normalized values, plus the user id and a server-side secret); the confirm handler recomputes the HMAC over the submitted payload and rejects the request with a recoverable error if the signatures do not match, before running normal validation. Equivalently-acceptable alternative: persist the pending confirmation payload server-side under a short-lived opaque token referenced by a single hidden form field, and reject any divergence between the token's stored payload and the submitted payload. Either approach must reject the four tamper cases in the acceptance criteria (modified amount, modified description, injected unknown `tagId`, swapped category id) before relying on revalidation alone.
   - **Atomicity at confirmation time**: confirmation creates/reuses categories, tags, and the resulting expense/recurring template in a single database transaction. If any step of the confirmation fails (unique-constraint race surfaced as a non-tag conflict, revalidation failure, unexpected DB error), nothing is partially persisted — no orphan category, tag, expense, or recurring template rows — and the user is returned to a recoverable error state with all values preserved.
   - **Validation-error preservation**: every re-render path — form validation errors, confirmation cancel, confirmation-time conflict, edit-form errors, recurring-form errors — preserves chip selections, `newTags` text, the new-category text, and all other field values.
   - **Safe rendering**: tag names, new-tag tokens, category names, and any other user-controlled value rendered into chip labels, hidden/preserved inputs, error messages, and confirmation pages must be HTML-escaped server-side (rely on JSX's default escaping; never `dangerouslySetInnerHTML`). Client JS must build optimistic chips with `textContent` and `setAttribute` only. **Safe rendering applies unconditionally to stored values, including legacy or otherwise-invalid tag/category names that violate the current `^[a-z0-9_-]{1,20}$` rule** (e.g. names predating the constraint, names introduced by direct DB mutation, or names produced by a prior bug) — the renderer never trusts that stored data matches current validation.
   - **Logging**: validation errors are not logged as server errors. Unexpected server errors log route + error class + stack; never log raw form bodies, query strings, descriptions, amounts, or tag/category names. Unexpected read failures during list/filter/Summary rendering produce the app's standard error response under the same logging discipline.

8. **Malformed Summary query handling**: handlers must never 500 on bad input. Unknown or syntactically invalid `tagId` values are silently dropped; invalid `dimension`, `granularity`, sort column name (including out-of-dimension columns per the dimension-aware allow-list above), or sort direction fall back to defaults; non-`YYYY-MM-DD` `from` / `to` values **and** shape-correct but invalid calendar dates (e.g. `2026-02-31`, `2026-13-01`, `2026-00-10`) are treated as absent. The same unknown-`tagId`-dropping applies to the list-page tag filter.

See PRD sections _Solution_, user stories 7–10 (entry), 14 (no-JS fallback), 31 (list filter), 37 (summary filter), 40 (year-included labels), 41 (chronological sort + tie-breakers), and _Implementation Decisions_ → _Summaries_ (Sort, time-period aggregation keys, malformed query parameters), _Forms and validation_, _Tag input contract (server-authoritative)_, _Safe rendering of user-controlled values_, _Error handling and logging_, and _Client JS (progressive enhancement)_.

### Acceptance criteria

#### Tag UI

- [ ] Given existing tags, when the user opens the expense entry form, then every tag is rendered as a chip-checkbox, alphabetically sorted (case-insensitive), wrapping to the viewport.
- [ ] Given the expense edit page and the recurring create/edit pages (and their confirmation routes), when the user views them, then they use the same chip-checkbox block + new-tag input as the entry form.
- [ ] Given the Summary page, when the user views the tag filter, then it is a chip-checkbox block (not a `<select multiple>`), reuses the shared component, has no new-tag input, and preserves AND semantics for two or more selected tags.
- [ ] Given the list-page filter bar and the entry form, when both are rendered, then they use the same shared chip-checkbox component (single source of truth); the list-page filter retains its existing AND/OR toggle and `mode=and|or` query-string contract unchanged.
- [ ] Given `/js/tag-chip-picker.js` is deleted, when any form page loads, then no 404 is logged and the chip UI works (with or without JS).
- [ ] Given `tag-chip-checkboxes.js` throws during init, when the user interacts with the form, then native checkboxes still toggle and the form still submits successfully.

#### Mutation validation

- [ ] Given the user toggles chips and submits, when the expense is created, then exactly the selected tags are attached, the server dedupes any repeated `tagId` values in the submitted body, and duplicates on the resulting expense are structurally impossible.
- [ ] Given a tampered submission with a `tagId` that does not exist or is syntactically invalid (not a ULID), when the mutation form is posted, then no DB lookup is performed for the bad id and the form re-renders with a recoverable global error and all other field values preserved (no 500).
- [ ] Given a mutation submission whose `tagId` count exceeds the configured cap (e.g. > 64), when posted, then the form re-renders with a recoverable validation error and all other values preserved.
- [ ] Given the user types new tag names in the adjacent text input and submits, when the confirmation page renders, then it lists every new tag (and any new category) and the full expense data; the server has split/trimmed/lowercased/deduped `newTags` (accepting any of `foo,bar` / `foo bar` / `foo, bar` / `foo,,bar` / `foo\nbar` separator forms), dropped empty tokens, and rejected any token failing `^[a-z0-9_-]{1,20}$` or any submission exceeding the raw-length / token-count caps with a recoverable error; confirming creates them atomically and attaches them.
- [ ] Given the user types a new-tag name that matches an existing tag (case-insensitively), when submitted, then the confirmation page does not list it as a new tag and the existing tag is attached exactly once.
- [ ] Given the user cancels the confirmation, or the form re-renders due to any validation error (entry, edit, recurring create/edit, recurring confirmation, expense confirmation), when the form re-renders, then chip selections, the new-tags text, the new-category text, and all other field values are preserved.

#### Confirmation / race behavior

- [ ] Given another request creates the same new tag between the initial submit and the confirmation submit (on either the expense or recurring confirmation flow), when the user confirms, then the existing tag is silently reused and attached without error.
- [ ] Given another request creates the same new category between the initial submit and the confirmation submit (on either the expense or recurring confirmation flow), when the user confirms, then the existing category is silently reused without error — the prior category race/collision handling remains intact.
- [ ] Given a tampered hidden field on the confirmation page (e.g. modified amount from `10.00` to a valid `20.00`, modified description, injected unknown `tagId`, or a valid `categoryId` swapped for a different valid `categoryId`), when confirmation is submitted, then the confirm handler rejects the tampered payload with a recoverable error via either an HMAC-signature mismatch over the canonical confirmation payload or a server-side pending-state divergence (whichever strategy the implementation chose) — revalidation alone is insufficient and hidden inputs are never trusted.
- [ ] Given any failure at confirmation time (unique-constraint surfaced as a non-tag conflict, revalidation failure, unexpected DB error), when confirmation aborts, then no partial expense, recurring template, category, or tag row is persisted, and the user is returned to a recoverable error state with all values preserved.

#### Summary sorting

- [ ] Given expenses across Jan–Apr of one year and default sort, when the Summary table renders with Month granularity, then rows appear in calendar order (`Jan 2026, Feb 2026, Mar 2026, Apr 2026`).
- [ ] Given expenses across all four calendar quarters of one year and default sort, when the Summary table renders with Quarter granularity, then rows appear `Jan-Mar 2026, Apr-Jun 2026, Jul-Sep 2026, Oct-Dec 2026`.
- [ ] Given expenses in `Dec 2025` and `Jan 2026` with the same category, when the Summary table renders, then two distinct rows appear (no cross-year aggregation) and `Dec 2025` sorts before `Jan 2026` in default ascending order.
- [ ] Given the user clicks the time-period column header, when the table re-renders, then sort toggles between chronological ascending and chronological descending using the internal `(year, monthIndex|quarterIndex)` key — never alphabetical of the rendered label.
- [ ] Given category + time-period combined sort, when the table renders, then category sorts case-insensitive alphabetical ascending and ties break on chronological time-period ascending.
- [ ] Given the user clicks `count`, `total`, `category`, or `tag`, when the table re-renders, then ties retain the default group/time ordering (group columns ascending, then chronological time-period ascending).
- [ ] Given a `sort` column not present in the current dimension's result table (e.g. `sort=tag` with dimension `Category` or `Time only`; `sort=category` with dimension `Tag` or `Time only`), when the page renders, then sort falls back to the default rather than erroring.
- [ ] Given a request to `/summary` with malformed parameters (unknown or syntactically-invalid `tagId`, unknown `dimension`, unknown `granularity`, unknown sort column or direction, non-`YYYY-MM-DD` `from`/`to`, or shape-correct but invalid calendar dates such as `2026-02-31`, `2026-13-01`, `2026-00-10`), when the page renders, then unknown values fall back to defaults (or are dropped, for `tagId`) and no 500 occurs. Stale `tagId` values are simply omitted from the rendered filter UI.
- [ ] Given an expense with no tags, when the Summary uses dimension `Tag` or `Category + Tag`, then the untagged expense contributes no rows; with dimension `Time only` or `Category`, it still contributes as expected.

#### Security and progressive enhancement

- [ ] Given any affected page or POST handler (entry, edit, recurring create/edit, expense and recurring confirmation routes, list, Summary), when an unauthenticated request is made, then it redirects to sign-in (auth gating preserved).
- [ ] Given a stored tag or category name containing HTML metacharacters (e.g. `<script>` — representing legacy data, direct DB mutation, or a prior bug, since current validation rejects such new names), when it is rendered as a chip label, in confirmation entries, in error messages, or as an optimistic JS chip, then the characters are escaped (server: JSX default; client: `textContent`) and no script executes — safe rendering applies unconditionally to stored values.
- [ ] Given a list-page or Summary filter request with excess `tagId` values beyond the configured cap or with syntactically-invalid ids, when the page renders, then the excess/invalid ids are silently dropped and no DB lookup is performed for them; the page still renders.
- [ ] Given JavaScript is disabled, when the user submits the entry form (or any other mutation form) with chip selections and new-tag text, then the confirmation page handles both correctly and the round-trip preserves all values.
